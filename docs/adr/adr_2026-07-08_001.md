# ADR-001: JIT internal design system in a pnpm/Turborepo monorepo

- **Status:** Accepted
- **Date:** 2026-07-08

---

## Context and problem statement

We are building a new internal design system (DS) that lives in the same repository as its consuming apps. There is **no npm registry involved and, currently, zero consumers** — the DS is being written now.

Three decisions must be made before the first component ships, because each is cheap to get right on a blank page and expensive-to-impossible to change once apps depend on the DS:

1. What monorepo tooling manages workspaces and task orchestration.
2. How apps consume the DS — as a compiled artifact or as raw source.
3. How the DS's public API boundary is defined and enforced.

The dominant force is **leverage from having zero consumers**: a strict public surface costs nothing today and becomes a breaking change the moment an app depends on it. "Tighten later" is a fantasy — later means consumers exist. So the bias is toward the strictest defensible setup now.

## Decision drivers

- **Config inheritance** — consumers should own Tailwind/PostCSS, JSX runtime, and compile target, not inherit the DS's build config.
- **Developer experience** — cross-package HMR and no build step in the DS during development.
- **Enforceable API boundary** — deep imports into component internals must be impossible, at both runtime and type-check time.
- **Scope reality** — FE-only, TypeScript, fewer than ~20 packages, fully internal. No polyglot, no external publishing.
- **Dev-build cost** — avoid a fat barrel that taxes every consumer's cold start and test run.

---

## Considered options

### Decision A — Monorepo tooling

| Option                    | Verdict             | Reasoning                                                                                                                                                                              |
| ------------------------- | ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **pnpm + Turborepo**      | **Chosen**          | FE-only, internal, small package count. Minimal config, free local caching, `catalog:` pins shared deps. Right default.                                                                |
| Nx                        | Rejected (deferred) | Platform-grade features (generators, enforced multi-team boundaries, `nx release`, polyglot, agentic CI context) exceed current need. Revisit if the repo goes polyglot or multi-team. |
| Bazel                     | Rejected            | Multi-language, thousands-of-packages tier. Not our scale.                                                                                                                             |
| npm/pnpm workspaces alone | Rejected            | No task graph or caching; compounding technical debt.                                                                                                                                  |

### Decision B — How apps consume the DS

| Option                           | Verdict    | Reasoning                                                                                                                                                                              |
| -------------------------------- | ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **JIT source (ship raw `.tsx`)** | **Chosen** | Consumer bundler transpiles on demand → config inheritance + real cross-package HMR + no DS build step. Valid only because all consumers are modern bundlers and nothing is published. |
| Compiled `dist` (ESM)            | Rejected   | Necessary only for external/registry consumers. Adds a build tax we don't need internally.                                                                                             |

> **Explicitly rejected rationale:** "JIT gives better tree-shaking." This is false. Compiled ESM with `preserveModules` + `sideEffects` tree-shakes identically. The real reasons are config inheritance and DX. Recorded here so it is never cited in review.

### Decision C — API boundary / encapsulation

| Option                                                          | Verdict             | Reasoning                                                                                                                                                                    |
| --------------------------------------------------------------- | ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`exports` allowlist, per-component subpaths, no root barrel** | **Chosen**          | The `exports` map is the wall — undeclared paths are unresolvable. Per-component subpaths keep each import's graph minimal. No root barrel avoids barrel-cost on dev builds. |
| Root `.` barrel (`./src/index.ts`)                              | Rejected (deferred) | Adding a barrel later is non-breaking; adding it now makes the barrel-cost path the default. Omit for now.                                                                   |
| Folder discipline / lint only                                   | Rejected as primary | A barrel or convention does **not** stop deep imports; only `exports` does. Lint is a backstop, not the wall.                                                                |

### Decision D — Type-level enforcement

`moduleResolution: "bundler"` in a shared `tsconfig.base.json` extended by the DS and every app, so a deep import fails `tsc`, not just the bundler. Legacy `node` resolution would enforce the boundary only at runtime — rejected.

---

## Decision

Adopt a **pnpm + Turborepo** monorepo in which the internal design system **ships raw TypeScript source (JIT)**, exposes a **per-component `exports` allowlist with no root barrel**, and is enforced at type level via a **shared `moduleResolution: "bundler"` tsconfig base**.

### Canonical implementation

**`packages/design-system/package.json`**

```jsonc
{
  "name": "@you/design-system",
  "private": true,
  "type": "module",
  "sideEffects": ["**/*.css"],
  "exports": {
    "./*": "./src/components/*/index.ts",
    "./tokens": "./src/tokens/index.ts",
    "./package.json": "./package.json",
  },
  "peerDependencies": { "react": "catalog:", "react-dom": "catalog:" },
}
```

**`tsconfig.base.json`** (extended by DS and every app)

```jsonc
{
  "compilerOptions": {
    "moduleResolution": "bundler",
    "module": "preserve",
    "target": "ES2022",
    "jsx": "react-jsx",
    "strict": true,
    "verbatimModuleSyntax": true,
    "isolatedModules": true,
  },
}
```

**`turbo.json`** — the DS has **no `build` task**; only apps build.

```jsonc
{
  "tasks": {
    "typecheck": { "dependsOn": ["^typecheck"] },
    "lint": {},
    "test": { "dependsOn": ["^typecheck"], "outputs": ["coverage/**"] },
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "!.next/cache/**"],
    },
    "dev": { "cache": false, "persistent": true },
  },
}
```

**`pnpm-workspace.yaml`**

```yaml
packages: ["packages/*", "apps/*"]
catalog:
  react: 19.0.0 # set to actual version
  react-dom: 19.0.0
```

**Folder convention** (`index.ts` is the only public face; siblings are unreachable):

```
src/components/Button/
  index.ts      # public
  Button.tsx    # internal
  Button.css    # internal
  internals.ts  # internal, never in exports
```

---

## Consequences

### Positive

- Deep imports (`@you/design-system/src/components/Button/internals`) fail at both resolution and `tsc` — the public surface is exactly the declared subpaths.
- Consumers inherit their own Tailwind/PostCSS/JSX/target config; the DS carries none.
- No DS build step; cross-package HMR in dev.
- Identical React everywhere via `catalog:` + peer dep, structurally preventing duplicate-React hook bugs.
- `sideEffects: ["**/*.css"]` from the first component — styles are never tree-shaken away.

### Negative / obligations on every consumer app

- **Must transpile the DS explicitly** — `transpilePackages: ["@you/design-system"]` in Next; equivalent for other bundlers. (Next does not transpile `node_modules` by default.)
- **Must whitelist the DS in the test runner** — Vitest `server.deps.inline` / Jest `transformIgnorePatterns`. The silent CI-red trap.
- **Typechecking is coupled** — consumers typecheck DS _source_, so a DS type error breaks every consumer's typecheck and DS generics add to everyone's `tsc` time. Keep DS types clean as a discipline.

### Risks

- A consumer that is **not** a modern bundler (plain Node, ts-node, bundler-less SSR) cannot consume raw source. Currently all consumers are Next apps; this decision must be revisited if that changes.

---

## Compliance / enforcement

1. **`exports` allowlist** — primary boundary (resolution + type level via `moduleResolution: bundler`).
2. **`eslint-plugin-import` `no-internal-modules`** — backstop for intra-package reaches the `exports` map cannot police (one component importing another's internals).
3. **Generated `exports` map** — a `scripts/gen-exports.mjs` run in pre-commit generates the map from `components/*/index.ts`, so exporting a component is a deliberate act, not a leaking default.

---

## Revisit when (supersession triggers)

- The DS gains an **external/registry consumer** → Decision B (JIT) must flip to compiled `dist`; supersede with a new ADR.
- A **non-bundler consumer** appears → same.
- The repo goes **polyglot** or needs enforced **multi-team boundaries + generators** → Decision A (tooling) reopens in favor of Nx.
- Barrel ergonomics become worth the cost → Decision C adds a root `.` barrel (non-breaking; amend rather than supersede).

---

## Follow-ups / notes

- **`react` as peer dependency** (not a regular dep) is a deliberate choice: the app owns React, the DS needs only a compatible one. Slightly more ceremony for an internal package, kept because it makes duplicate-React bugs structurally harder.
- **Placeholders to fill:** catalog React version; drop `optimizePackageImports` unless a root barrel is later added.
- **Splitting this ADR:** if you want each boundary individually revisable, carve **Decision B (JIT consumption)** and **Decision C (encapsulation)** into their own ADRs — they are the two most likely to be superseded, and standalone records make the `Superseded by` transition clean. Decisions A and D are stable enough to stay here.
