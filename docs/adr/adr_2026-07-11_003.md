# ADR-003: Accessibility testing strategy for the design system

- **Status:** Accepted (Layers 1 + 3 implemented; Layers 2 + 4 pending)
- **Date:** 2026-07-11
- **Relates to:** ADR-002 (shared chrome, focus ring, Radix wrappers)

---

## Context and problem statement

The kit is entering its Radix-wrapper phase (ADR-002): Checkbox, Switch, Select, Dialog, Menu, Tooltip — components whose whole value is keyboard, focus, and ARIA behavior. Accessibility is therefore a **product requirement**, not a nice-to-have, and it needs to be defended by tests that fail a build, not by manual inspection.

Today the a11y posture is:

- `@storybook/addon-a11y` is installed — but it only surfaces axe results in the **browser panel**. Nothing runs headless; a regression cannot turn CI red.
- Vitest + Testing Library + `jest-dom` cover behavior and class-token assertions in **jsdom**. jsdom has no layout or color engine, so it structurally cannot catch contrast or computed-style violations.
- Semantic color tokens are **TS source of truth** with explicit light + dark refs, but nothing asserts their contrast ratios — the reasoning in `CLAUDE.md` (e.g. "sky rejected because its step-9 needs dark text while `fg-on-info` is white") is documented, not enforced.

Net: accessibility is currently a manual, best-effort concern. As the interactive surface grows, that does not hold.

---

## Decision drivers

- **A11y regressions must fail CI**, not sit in a browser panel.
- **Reuse the story matrix** — variant × color × size × light/dark stories already are the coverage corpus; don't rebuild it.
- **Test at the right layer** — structural/ARIA in jsdom (fast), computed-style/contrast in a real browser or against tokens directly.
- **Protect documented token decisions** — contrast choices already reasoned about should be pinned.
- **Fit existing pipelines** — unit suite stays on every push; expensive browser work is on-demand until we promote it.
- **Low marginal cost per component** — new components inherit coverage from stories + shared assertions.

---

## Decision

Adopt a **layered** accessibility testing strategy. Each layer catches a class of defect the others cannot, ordered by leverage.

### Layer 1 — axe on every story (`@storybook/addon-vitest`) — primary ✅

Wire `@storybook/addon-a11y` into the Storybook **Vitest addon** (SB 10 path). Each story becomes a real-browser (Playwright) test with axe-core run against it, headless.

- Reuses the existing story matrix as the axe corpus — coverage comes for free from stories already maintained.
- Runs real computed styles → catches **contrast and computed-style** violations jsdom cannot.
- Violations fail `test:storybook` (on-demand workflow / local); not part of every-push unit CI.

**Shipped:** Vitest project `storybook` in `packages/ui/vitest.config.ts` (`@storybook/addon-vitest` + `@vitest/browser-playwright` + Playwright Chromium). Preview sets `parameters.a11y.test = "error"`. Foundation token grids opt out (`a11y.test: "off"`) — contrast for those is Layer 3.

**CI (on-demand):** `.github/workflows/a11y.yml` — `workflow_dispatch` only (not on every push/PR). Caches `~/.cache/ms-playwright` keyed by Playwright version; on cache hit only OS deps reinstall. Local / CI command: `pnpm --filter @design-system/ui test:storybook`. Default `pnpm test` stays unit-only so PR minutes stay cheap.

### Layer 2 — axe assertions in unit tests (direct `axe-core`)

For ad-hoc rendered trees and edge states not worth a full story (`aria-invalid` on Field/FormField error state, disabled Button, Spinner `role`/`aria-label`), run `axe.run(container)` in jsdom via a small local helper + matcher in `src/test/setup.ts`.

- Fast, jsdom, no browser.
- **Tooling:** call `axe-core` directly rather than a wrapper. `vitest-axe` is stalled (stuck at `0.1.0`, no stable `1.0`), and a `jest-axe` dependency reads oddly in a Vitest repo. `axe-core` is the maintained engine (already present transitively via Layer 1) and a ~15-line helper fits this repo's hand-rolled test-helper style (see `Button.test.tsx`). Own the thin layer instead of taking a questionable dep.
- **Scope caveat:** structural/ARIA only — jsdom cannot assert color contrast (that is Layer 1's job).

### Layer 3 — semantic-token contrast tests (unique to this architecture) ✅

A plain Vitest test resolves every semantic foreground/background pair (`fg-default` on `bg-canvas`, `fg-on-accent` on `accent-solid`, `fg-on-info` on the info solid, …) to hex and asserts the WCAG ratio (4.5:1 body text, 3:1 large text / UI) **in both light and dark**.

- No browser, no components — pure token math, mirrors the `generate-tokens.test.ts` invariant pattern.
- Pins the contrast decisions already documented in `CLAUDE.md`, so a token regen cannot silently break them.

**Shipped:** `packages/ui/src/tokens/colors/contrast.ts` + `contrast.test.ts` (unit project / every-push CI).

- Resolves palette refs via `@radix-ui/colors` (+ hand-authored `blue.css` for brand).
- Pair catalog `semanticContrastPairs` with roles:
  - **`body`** — min 4.5:1
  - **`large-ui`** — min 3:1 (Radix step-9 solids + white; body AA not claimed for small labels)
  - **`floor`** — known under-body-AA debt (soft status badges, `fg-subtle`, some solid-hovers); pin a measured floor so ratios cannot regress until tokens are deliberately improved
- Layer 1 Matrix axe failures map to this table — fix tokens by raising floors / promoting pairs to `body`, not by silencing stories.

### Layer 4 — keyboard/focus interaction coverage

`user-event` tab-order and `:focus-visible` ring assertions, expanding to Escape/arrow handling as menus/dialogs land. Prefer expressing these as **play functions** in stories so they double as living docs and are executed by Layer 1.

- Directly exercises the `focus-ring` util and Radix keyboard behavior that ADR-002 made the shared contract.

### Sequencing

1. **Layer 1** first — highest coverage-per-effort; headless story axe runnable locally and via on-demand workflow.
2. **Layer 3** next — natural extension of the token pipeline; protects reasoned decisions.
3. **Layers 2 and 4** incrementally, per component.

### Non-goals

- Manual screen-reader QA scripts — valuable but out of scope for automated gating.
- Running Layer 1 browser tests on every push/PR — too expensive for GitHub Actions minutes right now; use the on-demand `A11y (Storybook axe)` workflow (and later promote to required checks when warranted).
- Replacing the addon-a11y browser panel — it stays for authoring; Layer 1 is its headless counterpart.
- Full WCAG 2.2 AAA — target is AA.

---

## Consequences

### Positive

- Accessibility regressions fail CI instead of hiding in a browser panel.
- New components inherit a11y coverage from the stories they already ship.
- Contrast decisions in `CLAUDE.md` become enforced invariants.
- Real-browser axe fills the gap jsdom structurally cannot.

### Obligations

- New interactive components ship stories complete enough for Layer 1 to be meaningful (states, not just the happy path).
- Run the on-demand a11y workflow (or `test:storybook` locally) when shipping interactive/story changes — Layer 1 is not a free PR gate until it is promoted.
- Playwright browsers are cached in CI by package version; bumping `playwright` invalidates the cache intentionally.
- The token-contrast test must be updated deliberately when semantic pairs change — that is the point.

### Risks

- Story-driven axe only covers what stories render; thin story matrices mean thin coverage — code review should treat missing state stories as a gap.
- On-demand Layer 1 can be skipped by habit — treat running it before merge of UI PRs as process until it becomes a required check.

---

## Compliance / enforcement

1. **Layer 1** via on-demand workflow `A11y (Storybook axe)` (`workflow_dispatch`) and local `test:storybook` — not part of every-push CI yet.
2. **Layer 3** token-contrast tests run with the standard unit suite (`pnpm test` / `contrast.test.ts`).
3. **Code review** against story completeness and the ADR-002 focus-ring / `ControlSize` rules.
4. **addon-a11y panel** remains available for authoring-time checks.

---

## Revisit when

- First Radix wrapper lands → confirm Layer 4 play-function pattern on a real keyboard-driven control.
- Ready to spend Actions minutes on every PR → promote `a11y.yml` to `pull_request` / required check (cache already in place).
- Target moves beyond WCAG AA → revisit thresholds in Layer 3 and axe rules.
