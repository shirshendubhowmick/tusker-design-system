# ADR-002: Shared chrome before Radix-backed components

- **Status:** Accepted (foundations delivered)
- **Date:** 2026-07-11
- **Supersedes:** `docs/plan.md` (living build plan; removed)

---

## Context and problem statement

After ADR-001 (monorepo + JIT + exports) and an initial form/action kit (`Button`, `IconButton`, `Input`, `FormField`, `Text`, `Spinner`), the next interactive components (Checkbox, Switch, Select, Dialog, Menu, Tooltip, …) were expected to wrap **Radix UI primitives** for keyboard, focus management, and a11y behavior.

If each Radix wrapper invented its own focus ring, panel chrome, scrim, or form layout, the kit would fragment: inconsistent focus language, duplicated CVA blocks, and hard-to-restyle overlays. Those shared pieces are cheap to extract once and expensive to retrofit across many wrappers.

A living `docs/plan.md` sequenced the work. That plan is **complete for its original scope** (shared layer before the first Radix control / overlay). This ADR records the decisions and what shipped so implementers do not re-open the foundation questions.

---

## Decision drivers

- **Thin styled Radix re-exports** — behavior + public API from Radix; look and tokens from the DS.
- **One focus language** — every interactive control uses the same ring geometry and intent tints.
- **One floating chrome language** — menus, dialogs, popovers share surface + scrim recipes and z-index tiers.
- **Form layout not tied to text inputs** — choice controls (checkbox / switch / radio) need horizontal label rows without forking `FormField`.
- **Semantic tokens only** — components never reach raw palette steps.
- **JIT public surface** — each util/component is a deliberate `exports` subpath (ADR-001).

---

## Decision

**Build and stabilize shared chrome and composition primitives before shipping Radix-backed components.** Radix packages are not introduced until wrappers can depend on these recipes rather than inventing local styles.

### What was delivered

| Area                | Public API                     | Role                                                                                                                                                                                                               |
| ------------------- | ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Focus ring          | `@design-system/ui/focus-ring` | `focusRing()` / `focusRingClass` — self (`:focus-visible`) vs within (`:focus-within`), intent tints (default / success / danger / warning), same 3px geometry as `shadow-focus`. Wired into `Button` and `Input`. |
| Surface             | `@design-system/ui/surface`    | `surface()` / `surfaceClass` — bg, border, radius, elevation, padding. Recipes: `card`, `popover`, `menu`, `dialog`, `inverse`. Section padding via `surfaceSectionClass`.                                         |
| Overlay / scrim     | `@design-system/ui/overlay`    | `overlay()` / `overlayClass.scrim` (`fixed inset-0`, `z-overlay`, `bg-overlay-scrim`); `overlayClass.catch` for transparent outside-click layers.                                                                  |
| Field layout        | `@design-system/ui/Field`      | Orientation-aware form row: vertical (label above) vs horizontal (choice row). Description + message (tone), merges `id` / `aria-describedby` / `aria-invalid` onto a single control child.                        |
| Label               | `@design-system/ui/Label`      | Thin label primitive (`text-label-md`, required marker, disabled). Composed by `Field` / `FormField`.                                                                                                              |
| Form convenience    | `@design-system/ui/FormField`  | Thin vertical `Field` + `Input` — not the only form layout path.                                                                                                                                                   |
| Badge               | `@design-system/ui/Badge`      | Soft / outline / solid × semantic colors × sm/md.                                                                                                                                                                  |
| Links               | Global CSS in `styles.css`     | No separate `Link` component — product anchor styles + focus ring on `a`.                                                                                                                                          |
| Polymorphic actions | `Button` `as` prop             | Button-looking navigation via `"a"` or router `Link`.                                                                                                                                                              |
| Control scale       | tokens `ControlSize`           | Shared `sm` / `md` / `lg` heights, box lock, glyphs — already present; remains the sizing contract for interactive targets.                                                                                        |
| Z-index             | tokens z-index scale           | `base` → `raised` → `dropdown` → `sticky` → `overlay` → `modal` → `toast` → `tooltip` — ready for floating layers.                                                                                                 |

### Explicit non-goals of this workstream

- Shipping Radix wrappers (Checkbox, Dialog, Select, …) — **next** workstream, not blocked on further foundation invention beyond the small gaps below.
- A separate `Link` component — rejected in favor of global anchor styles + polymorphic `Button`.
- Rebuilding tokens or monorepo infra — already covered by ADR-001.

### Small gap left open (not blocking)

| Item          | Notes                                                                                                                     |
| ------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Separator** | Still optional CSS-only or thin Radix Separator when menus/dialogs need dividers. Can land with the first menu/dialog PR. |

### Radix dependency + public API (decided)

Applies to every `@radix-ui/react-*` **primitive** used for interactive controls/overlays (Checkbox, Dialog, Select, …). Distinct from `@radix-ui/colors` (already a runtime dep) and optional `@radix-ui/react-icons` (peer).

| Rule                           | Decision                                                                                                                                                                                                                             |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **npm placement**              | **`dependencies` of `@design-system/ui`**, not peers. Consumers install only `@design-system/ui` (+ existing peers: React, CVA). They never add `@radix-ui/react-*` themselves.                                                      |
| **Version ownership**          | DS pins and upgrades Radix. App trees get one copy via the DS package graph (JIT source still resolves through the workspace).                                                                                                       |
| **Public import**              | From the design system only — e.g. `@design-system/ui/Checkbox`, never `@radix-ui/react-checkbox` in app code.                                                                                                                       |
| **API contract**               | **Radix’s API unchanged.** Parts, props, ref/`asChild` behavior match upstream. Consumers use [Radix docs](https://www.radix-ui.com/primitives/docs/overview/introduction) for behavior; they only change the import path.           |
| **What the DS adds**           | Default **styling** (semantic tokens, `focusRing`, `surface`/`overlay`, `ControlSize` where applicable) via `className` / CVA on the same parts. No renamed props, no replaced state model, no DS-only required props for basic use. |
| **Export shape**               | Re-export the primitive surface **as-is** (same part names / composition model as upstream). Style inside the wrapper modules; do not invent a parallel product prop API for primitives.                                             |
| **Composition with DS layout** | Higher-level kit pieces (`Field`, `FormField`, …) may **compose** primitives; that composition is separate from the primitive export, which stays Radix-faithful.                                                                    |

Rationale: one install surface for apps; zero Radix version drift across consumers; documentation leverage (upstream docs stay valid); DS still owns look via shared chrome.

## Working rules (binding for implementers)

These remain the contract for new components, including future Radix primitives:

1. **Semantic tokens only** in components (`bg-accent-solid`, `text-fg-muted`) — never raw palette steps.
2. **`ControlSize`** for interactive hit targets (`sm` / `md` / `lg`) — do not re-encode `h-8` / `h-9` / `h-10` ad hoc.
3. **Focus rings** via `focusRing()` / `focusRingClass` — do not invent per-component ring shadows.
4. **Floating chrome** via `surface()` / `overlay()` — do not invent per-component panel/scrim classes.
5. **Radix owns behavior and the primitive public API** (keyboard, focus trap, a11y, props/parts). **DS owns look only** on those primitives — style, do not re-shape the contract. (See “Radix dependency + public API” above.)
6. **Colocate** tests + Storybook matrix (variant × size × state).
7. **JIT export** — new public component = `src/components/<Name>/index.ts` + `pnpm exports:generate`.
8. Prefer **thin styled re-exports**; push shared chrome into utils/recipes, not copy-paste CVA blocks.

### Focus ring intent is deliberately not mapped to `Button` `color`

`focusRing()` exposes intent tints (`default` / `success` / `danger` / `warning`), and
`Input` maps them from its `color` variant — there, `color` is the field's **validation
state**, the ring reinforces feedback the user must notice, and the field surface is always
neutral (`bg-bg-surface`) so a tinted ring keeps good contrast against its chrome.

`Button` intentionally uses `focusRing()` (brand `default`) for **every** `color`. On a
button, `color` is **action intent** (`danger`, `success`, …), already fully expressed by
the fill / border / text — a tinted ring adds no information. Two concrete reasons not to
map it:

- **Focus visibility (WCAG 2.4.7 / 1.4.11).** A `primary` `danger` button is
  `bg-danger-solid`; a `shadow-focus-danger` ring would be the same hue as the fill and
  nearly vanish against it. `Input` never hits this because its surface is neutral. A single
  brand ring keeps ring-to-control contrast predictable across all variants and colors.
- **Semantics.** A red ring reads as "this control is in an error state" — correct on an
  errored `Input`, misleading on a destructive `Button`. Keyboard focus is wayfinding
  ("where am I"); one consistent brand ring is the feature, not an inconsistency to fix.

If product ever wants tinted button rings, the only defensible scope is `secondary` /
`tertiary` (neutral-surface) buttons via `compoundVariants` + `focusRing({ intent })`,
keeping `primary` solid buttons on the brand ring for the contrast reason above. Treat
this as a deliberate split, not drift.

| Before shipping…                                 | Requires                                                                     |
| ------------------------------------------------ | ---------------------------------------------------------------------------- |
| First choice control (Checkbox / Switch / Radio) | Focus ring + `Field` horizontal layout + `Label` + `ControlSize` — **ready** |
| First overlay (Dialog / Drawer)                  | Surface + scrim + z-index tiers — **ready**                                  |
| Select / Menu content                            | Surface + focus on trigger — **ready**                                       |

Suggested product order (not a living backlog — product priority may reorder):

Checkbox → Switch → Radio Group → Select → Alert (may be non-Radix) → Dialog → Tooltip → Dropdown Menu → Toast.

Defer full Table/DataList, date pickers, app chrome, and charts until the primitives above exist where needed.

## Consequences

### Positive

- Radix PRs stay thin: style upstream parts with existing recipes; no parallel prop API to maintain.
- Consumers read Radix docs and import from `@design-system/ui/*` — one mental model for behavior.
- Apps never install or pin `@radix-ui/react-*`; no multi-version Radix drift.
- Focus, elevation, and form layout stay consistent across the kit.
- `FormField` no longer owns all form UX — choice and custom controls share `Field`.
- Public API stays intentional via generated `exports` (focus-ring, surface, overlay, Field, Label, Badge, …).

### Obligations

- New interactive components **must** use `focusRing` / `ControlSize` (and surface/overlay when floating) unless there is a documented exception.
- Each new `@radix-ui/react-*` package is a **`dependencies`** entry on `@design-system/ui`, exported via JIT subpath with **unchanged** Radix API + DS default styles.
- App code must not import `@radix-ui/react-*` directly for kit primitives (icons remain the optional-peer exception).
- Separator still needs a home before dense menus look finished — acceptable to co-ship with Menu/Dialog.

### Risks

- If a wrapper bypasses recipes “just this once,” drift returns; code review should treat local focus/panel CSS as a smell.
- If a wrapper **renames props or drops parts**, consumers lose Radix-doc fidelity — treat API drift as a review blocker.
- DS owns Radix upgrades: a major Radix bump is a DS release concern (changelog / codemods for consumers only if upstream broke its own API).
- Global anchor styles affect every consumer that imports `styles.css` — intentional product choice; apps that need unstyled links must opt out locally.

---

## Compliance / enforcement

1. **Code review** against the working rules above.
2. **`exports` map** — shared utils are first-class subpaths; deep imports remain blocked (ADR-001).
3. **Colocated tests** for utils (`focus-ring`, `surface`, `overlay`) and layout components (`Field`, `Label`, `FormField`, `Badge`).
4. **Storybook** foundations + component stories for the shipped kit.

---

## Revisit when

- A consumer truly needs an unstyled / alternate Radix tree (unlikely in this monorepo) → re-evaluate dependency-vs-peer only with a strong reason.
- Upstream Radix breaks composition patterns we rely on for default styling → may need a thin mapping layer (still prefer keeping public names aligned).
- A consumer needs unstyled anchors by default → re-evaluate global `a` styles vs a `Link` primitive.
- External/registry packaging → orthogonal; follow ADR-001 supersession for JIT.
