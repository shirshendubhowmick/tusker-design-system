# ADR-002: Shared chrome before Radix-backed components

- **Status:** Accepted (foundations delivered)
- **Date:** 2026-07-11
- **Supersedes:** `docs/plan.md` (living build plan; removed)

---

## Context and problem statement

After ADR-001 (monorepo + JIT + exports) and an initial form/action kit (`Button`, `IconButton`, `Input`, `FormField`, `Text`, `Spinner`), the next interactive components (Checkbox, Switch, Select, Dialog, Menu, Tooltip, ‚Ä¶) were expected to wrap **Radix UI primitives** for keyboard, focus management, and a11y behavior.

If each Radix wrapper invented its own focus ring, panel chrome, scrim, or form layout, the kit would fragment: inconsistent focus language, duplicated CVA blocks, and hard-to-restyle overlays. Those shared pieces are cheap to extract once and expensive to retrofit across many wrappers.

A living `docs/plan.md` sequenced the work. That plan is **complete for its original scope** (shared layer before the first Radix control / overlay). This ADR records the decisions and what shipped so implementers do not re-open the foundation questions.

---

## Decision drivers

- **Thin Radix wrappers** ‚Äî behavior from Radix; look, tokens, and product API from the DS.
- **One focus language** ‚Äî every interactive control uses the same ring geometry and intent tints.
- **One floating chrome language** ‚Äî menus, dialogs, popovers share surface + scrim recipes and z-index tiers.
- **Form layout not tied to text inputs** ‚Äî choice controls (checkbox / switch / radio) need horizontal label rows without forking `FormField`.
- **Semantic tokens only** ‚Äî components never reach raw palette steps.
- **JIT public surface** ‚Äî each util/component is a deliberate `exports` subpath (ADR-001).

---

## Decision

**Build and stabilize shared chrome and composition primitives before shipping Radix-backed components.** Radix packages are not introduced until wrappers can depend on these recipes rather than inventing local styles.

### What was delivered

| Area                | Public API                     | Role                                                                                                                                                                                                               |
| ------------------- | ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Focus ring          | `@design-system/ui/focus-ring` | `focusRing()` / `focusRingClass` ‚Äî self (`:focus-visible`) vs within (`:focus-within`), intent tints (default / success / danger / warning), same 3px geometry as `shadow-focus`. Wired into `Button` and `Input`. |
| Surface             | `@design-system/ui/surface`    | `surface()` / `surfaceClass` ‚Äî bg, border, radius, elevation, padding. Recipes: `card`, `popover`, `menu`, `dialog`, `inverse`. Section padding via `surfaceSectionClass`.                                         |
| Overlay / scrim     | `@design-system/ui/overlay`    | `overlay()` / `overlayClass.scrim` (`fixed inset-0`, `z-overlay`, `bg-overlay-scrim`); `overlayClass.catch` for transparent outside-click layers.                                                                  |
| Field layout        | `@design-system/ui/Field`      | Orientation-aware form row: vertical (label above) vs horizontal (choice row). Description + message (tone), merges `id` / `aria-describedby` / `aria-invalid` onto a single control child.                        |
| Label               | `@design-system/ui/Label`      | Thin label primitive (`text-label-md`, required marker, disabled). Composed by `Field` / `FormField`.                                                                                                              |
| Form convenience    | `@design-system/ui/FormField`  | Thin vertical `Field` + `Input` ‚Äî not the only form layout path.                                                                                                                                                   |
| Badge               | `@design-system/ui/Badge`      | Soft / outline / solid √ó semantic colors √ó sm/md.                                                                                                                                                                  |
| Links               | Global CSS in `styles.css`     | No separate `Link` component ‚Äî product anchor styles + focus ring on `a`.                                                                                                                                          |
| Polymorphic actions | `Button` `as` prop             | Button-looking navigation via `"a"` or router `Link`.                                                                                                                                                              |
| Control scale       | tokens `ControlSize`           | Shared `sm` / `md` / `lg` heights, box lock, glyphs ‚Äî already present; remains the sizing contract for interactive targets.                                                                                        |
| Z-index             | tokens z-index scale           | `base` ‚Üí `raised` ‚Üí `dropdown` ‚Üí `sticky` ‚Üí `overlay` ‚Üí `modal` ‚Üí `toast` ‚Üí `tooltip` ‚Äî ready for floating layers.                                                                                                 |

### Explicit non-goals of this workstream

- Shipping Radix wrappers (Checkbox, Dialog, Select, ‚Ä¶) ‚Äî **next** workstream, not blocked on further foundation invention beyond the small gaps below.
- A separate `Link` component ‚Äî rejected in favor of global anchor styles + polymorphic `Button`.
- Rebuilding tokens or monorepo infra ‚Äî already covered by ADR-001.

### Small gap left open (not blocking)

| Item                                 | Notes                                                                                                                                                                                                                                                         |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Separator**                        | Still optional CSS-only or thin Radix Separator when menus/dialogs need dividers. Can land with the first menu/dialog PR.                                                                                                                                     |
| **Radix peer / wrapper conventions** | Document when adding the first `@radix-ui/react-*` package: prefer **peerDependencies**, pin policy, re-export/compose behavior only; own styles, tokens, `cn`/`cva`, and DS props. Capture in `packages/ui/README.md` or a short follow-up ADR at that time. |

## Working rules (binding for implementers)

These remain the contract for new components, including future Radix wrappers:

1. **Semantic tokens only** in components (`bg-accent-solid`, `text-fg-muted`) ‚Äî never raw palette steps.
2. **`ControlSize`** for interactive hit targets (`sm` / `md` / `lg`) ‚Äî do not re-encode `h-8` / `h-9` / `h-10` ad hoc.
3. **Focus rings** via `focusRing()` / `focusRingClass` ‚Äî do not invent per-component ring shadows.
4. **Floating chrome** via `surface()` / `overlay()` ‚Äî do not invent per-component panel/scrim classes.
5. **Radix owns behavior** (keyboard, focus trap, a11y primitives); DS owns look and product API.
6. **Colocate** tests + Storybook matrix (variant √ó size √ó state).
7. **JIT export** ‚Äî new public component = `src/components/<Name>/index.ts` + `pnpm exports:generate`.
8. Prefer **thin wrappers**; push shared chrome into utils/recipes, not copy-paste CVA blocks.

### Minimum dependencies for next components

| Before shipping‚Ä¶                                 | Requires                                                                     |
| ------------------------------------------------ | ---------------------------------------------------------------------------- |
| First choice control (Checkbox / Switch / Radio) | Focus ring + `Field` horizontal layout + `Label` + `ControlSize` ‚Äî **ready** |
| First overlay (Dialog / Drawer)                  | Surface + scrim + z-index tiers ‚Äî **ready**                                  |
| Select / Menu content                            | Surface + focus on trigger ‚Äî **ready**                                       |
| Tooltip                                          | z-tooltip (+ light surface-ish chrome) ‚Äî **ready**                           |

Suggested product order (not a living backlog ‚Äî product priority may reorder):

Checkbox ‚Üí Switch ‚Üí Radio Group ‚Üí Select ‚Üí Alert (may be non-Radix) ‚Üí Dialog ‚Üí Tooltip ‚Üí Dropdown Menu ‚Üí Toast.

Defer full Table/DataList, date pickers, app chrome, and charts until the primitives above exist where needed.

## Consequences

### Positive

- Radix PRs can stay thin: compose Radix parts + apply existing recipes.
- Focus, elevation, and form layout stay consistent across the kit.
- `FormField` no longer owns all form UX ‚Äî choice and custom controls share `Field`.
- Public API stays intentional via generated `exports` (focus-ring, surface, overlay, Field, Label, Badge, ‚Ä¶).

### Obligations

- New interactive components **must** use `focusRing` / `ControlSize` (and surface/overlay when floating) unless there is a documented exception.
- Adding a Radix dependency requires settling peer vs runtime dep and documenting the wrapper pattern once (see gap above).
- Separator still needs a home before dense menus look finished ‚Äî acceptable to co-ship with Menu/Dialog.

### Risks

- If a wrapper bypasses recipes ‚Äújust this once,‚Äù drift returns; code review should treat local focus/panel CSS as a smell.
- Global anchor styles affect every consumer that imports `styles.css` ‚Äî intentional product choice; apps that need unstyled links must opt out locally.

---

## Compliance / enforcement

1. **Code review** against the working rules above.
2. **`exports` map** ‚Äî shared utils are first-class subpaths; deep imports remain blocked (ADR-001).
3. **Colocated tests** for utils (`focus-ring`, `surface`, `overlay`) and layout components (`Field`, `Label`, `FormField`, `Badge`).
4. **Storybook** foundations + component stories for the shipped kit.

---

## Revisit when

- First `@radix-ui/react-*` package is added ‚Üí document peer/wrapper conventions (amend this ADR or add ADR-003).
- A consumer needs unstyled anchors by default ‚Üí re-evaluate global `a` styles vs a `Link` primitive.
- External/registry packaging ‚Üí orthogonal; follow ADR-001 supersession for JIT.
